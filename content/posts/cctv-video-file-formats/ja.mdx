CCTV映像に適切なファイル形式を選択することは、映像品質、ストレージ要件、再生互換性、フォレンジックでの使用性に影響します。このガイドでは、監視システムで使用される主要なコンテナ形式とコーデックを解説します。

## コンテナとコーデック

[コンテナ形式](https://en.wikipedia.org/wiki/Container_format)（MP4やAVIなど）は動画、音声、メタデータを一つにまとめます。[コーデック](https://en.wikipedia.org/wiki/Video_codec)（H.264やH.265など）はそのコンテナ内で動画がどのように圧縮されるかを決定します。最新のCCTVシステムは通常、MP4コンテナ内でH.264またはH.265動画を使用します。

### AVI（Audio Video Interleave）

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#6366f1" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <rect x="34" y="36" width="8" height="4" rx="1" fill="#6366f1" opacity="0.2"/>

[AVI](https://en.wikipedia.org/wiki/Audio_Video_Interleave)は1992年にMicrosoftによって導入されました。最小限の圧縮でデータを保存し、高品質で大きなファイルを生成します。

**最適な用途:** 詳細がストレージより重要な短期間のフォレンジッククリップ。

| 長所 | 短所 |
|------|------|
| 高品質、広い再生サポート | 非常に大きなファイルサイズ |
| 複雑なエンコード不要 | 圧縮効率が低い |

### MP4（MPEG-4 Part 14）

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#10b981" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <rect x="6" y="6" width="36" height="36" rx="4"/>
  <rect x="10" y="10" width="28" height="28" rx="2" fill="#10b981" opacity="0.1"/>
  <polygon points="20,16 20,32 32,24" fill="#10b981" opacity="0.4"/>
</svg>

[MP4](https://en.wikipedia.org/wiki/MP4_file_format)は近代的な監視システムで支配的な形式です。圧縮効率と、NVR、モバイルアプリ、クラウドプラットフォーム間の幅広い互換性のバランスを取ります。

**最適な用途:** 長期録画、リモートビューイング、マルチプラットフォームワークフロー。

| 長所 | 短所 |
|------|------|
| 優れた圧縮対品質比 | AVIよりやや複雑なエンコード |
| ユニバーサルデバイスサポート | |
| ストリーミング性能良好 | |

### MOV（QuickTime File Format）

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#a855f7" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <rect x="6" y="6" width="36" height="36" rx="6"/>
  <path d="M16 14 L16 34 L34 24 Z" fill="#a855f7" opacity="0.3"/>
  <circle cx="14" cy="14" r="2" fill="#a855f7" opacity="0.4"/>
</svg>

[MOV](https://en.wikipedia.org/wiki/QuickTime_File_Format)はAppleによって開発され、ポストプロダクションワークフローで一般的です。一部のNVRは編集やアーカイブ目的でMOVにエクスポートします。

**最適な用途:** macOSベースのワークフローやビデオ編集用の映像。

| 長所 | 短所 |
|------|------|
| 高品質、編集に適したコーデックをサポート | MP4より大きい |
| メタデータサポート良好 | Appleエコシステム外でのネイティブサポートが少ない |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#f59e0b" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MKV](https://en.wikipedia.org/wiki/Matroska)はオープンスタンダードのコンテナで、1つのファイルに複数のオーディオトラック、字幕ストリーム、チャプターマーカーを保持できます。

**最適な用途:** マルチカメラのエクスポートやメタデータ付き映像のアーカイブ。

| 長所 | 短所 |
|------|------|
| 1ファイルに無制限のトラックをサポート | 主流のNVRソフトウェアでのサポートが少ない |
| オープンソース、ライセンス料不要 | 汎用的な互換性がない |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#3b82f6" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[WMV](https://en.wikipedia.org/wiki/Windows_Media_Video)はMicrosoftによってWindowsベースの再生用に設計されました。小さなファイルサイズを実現しますが、品質が著しく低下します。

**最適な用途:** レガシーなWindows専用DVRシステム。

| 長所 | 短所 |
|------|------|
| 小さなファイルサイズ | 最新コーデックより低品質 |
| ネイティブWindows再生 | クロスプラットフォームサポートが貧弱 |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#ef4444" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[FLV](https://en.wikipedia.org/wiki/Flash_Video)は2000年代にWebストリーミングで広く使用されました。Adobeが2020年にFlashサポートを終了して以来、ほとんどが非推奨です。

**最適な用途:** レガシーなWebカメラやブラウザベースの監視インターフェース。

| 長所 | 短所 |
|------|------|
| ファイルが小さく、ストリーミングが高速 | 時代遅れの技術 |
| | 最新の互換性が非常に限定的 |

### H.264（Advanced Video Coding）

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <text x="18" y="29" fontSize="12" fill="#059669" fontWeight="bold" fontFamily="monospace">264</text>

[H.264](https://en.wikipedia.org/wiki/H.264/MPEG-4_AVC)はCCTVで最も広く採用されている動画圧縮標準です。同等の品質を維持しながら、MJPEGと比較してファイルサイズを最大50%削減できます。

**最適な用途:** 既存のハードウェアでのHD録画と24時間365日の継続的な監視。

| 長所 | 短所 |
|------|------|
| 幅広いハードウェアデコーダーサポート | ライセンス制約あり（ほとんどのOEMがカバー） |
| 成熟しており、監視で十分にテスト済み | H.265に置き換えられつつある |

### H.265（High Efficiency Video Coding）

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <circle cx="18" cy="24" r="12" opacity="0.6"/>

  <text x="13" y="29" fontSize="10" fill="#059669" fontWeight="bold" fontFamily="monospace">265</text>
  <text x="25" y="29" fontSize="10" fill="#059669" fontWeight="bold" fontFamily="monospace">265</text>

[H.265](https://en.wikipedia.org/wiki/High_Efficiency_Video_Coding)（HEVCとも呼ばれる）はH.264の圧縮率を約2倍にし、4KおよびマルチチャンネルNVRの最適な選択肢となっています。

**最適な用途:** 4Kカメラ、高チャンネル数NVR、ストレージや帯域幅が限られたシステム。

| 長所 | 短所 |
|------|------|
| 同じ品質でH.264より約50%小さいファイル | より多くの処理能力が必要 |
| 4K+解像度に最適 | 古いNVRハードウェアでのサポートが限定的 |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MJPEG](https://en.wikipedia.org/wiki/Motion_JPEG)は各フレームを独立したJPEG画像としてエンコードします。これによりフレームごとの分析が簡単になりますが、フレーム間圧縮を使用しないため大きなファイルになります。

**最適な用途:** 低レイテンシエンコードや簡単なフレーム抽出が必要なアプリケーション。

| 長所 | 短所 |
|------|------|
| 各フレームが完全な画像（静止画の取得が容易） | H.264/H.265と比較して圧縮が不十分 |
| 低エンコードレイテンシ | 連続録画でのファイルサイズが大きい |

### RAW / 非圧縮動画

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[非圧縮動画](https://en.wikipedia.org/wiki/Uncompressed_video)はセンサーからすべてのピクセルを圧縮なしでキャプチャします。ファイルは巨大ですが、最大の証拠価値を提供します。

**最適な用途:** フォレンジック証拠、法廷提出物、調査分析。

| 長所 | 短所 |
|------|------|
| 圧縮による劣化なし | 非現実的に大きなストレージ要件 |
| 分析のための最大の詳細 | 専用の再生ソフトウェアが必要 |

多くのDVRメーカーは、タイムスタンプ、動体イベント、カメラIDなどのメタデータを埋め込む[プロプライエタリ](https://en.wikipedia.org/wiki/Proprietary_format)コンテナ形式を使用しています。これらのファイルの再生にはベンダーのソフトウェアまたはプレーヤーが必要です。

**最適な用途:** メーカーのエコシステム内でのシームレスな再生。

| 長所 | 短所 |

| 豊富なメタデータ（時間、動き、カメラID） | 単一ベンダーに依存 |
| 特定のハードウェアに最適化 | エクスポートには通常変換が必要 |

## ストレージの考慮事項

<svg viewBox="0 0 64 64" width="48" height="48" fill="none" stroke="#7c3aed" strokeWidth="2" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'12px'}}>

コーデックの選択はストレージ計画に直接影響します。24時間365日録画する4チャンネル1080pシステムでは、およそ以下の容量が必要です：

| コーデック | 1日あたりのストレージ（4×1080p @ 15fps） |
|-------|-----------------------------------|
| MJPEG | ~300 GB |
| H.264 | ~80 GB |
| H.265 | ~40 GB |
| RAW   | ~2+ TB |

これらのコーデックの技術的な詳細については、[H.264参照実装](https://en.wikipedia.org/wiki/X264)または[x265プロジェクト](https://en.wikipedia.org/wiki/X265)を参照してください。
