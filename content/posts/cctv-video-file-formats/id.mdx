Memilih format file yang tepat untuk rekaman CCTV memengaruhi kualitas video, kebutuhan penyimpanan, kompatibilitas pemutaran, dan kegunaan forensik. Panduan ini mencakup format kontainer dan codec utama yang digunakan dalam sistem pengawasan.

## Kontainer vs Codec

<svg viewBox="0 0 64 64" width="48" height="48" fill="none" stroke="#2563eb" strokeWidth="2" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'12px'}}>

[Format kontainer](https://en.wikipedia.org/wiki/Container_format) (seperti MP4 atau AVI) membungkus video, audio, dan metadata bersama-sama. [Codec](https://en.wikipedia.org/wiki/Video_codec) (seperti H.264 atau H.265) menentukan bagaimana video dikompresi di dalam kontainer tersebut. Sistem CCTV modern biasanya menggunakan video H.264 atau H.265 di dalam kontainer MP4.

### AVI (Audio Video Interleave)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#6366f1" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <rect x="34" y="36" width="8" height="4" rx="1" fill="#6366f1" opacity="0.2"/>

[AVI](https://en.wikipedia.org/wiki/Audio_Video_Interleave) diperkenalkan oleh Microsoft pada tahun 1992. Format ini menyimpan data dengan kompresi minimal, menghasilkan file besar dengan kualitas tinggi.

**Terbaik untuk:** Klip forensik jangka pendek yang mengutamakan detail daripada penyimpanan.

| Kelebihan | Kekurangan |
|------|------|
| Kualitas tinggi, dukungan pemutaran luas | Ukuran file sangat besar |
| Tidak perlu encoding kompleks | Efisiensi kompresi buruk |

### MP4 (MPEG-4 Part 14)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#10b981" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MP4](https://en.wikipedia.org/wiki/MP4_file_format) adalah format dominan dalam pengawasan modern. Format ini menyeimbangkan efisiensi kompresi dengan kompatibilitas luas di seluruh NVR, aplikasi seluler, dan platform cloud.

**Terbaik untuk:** Rekaman jangka panjang, peninjauan jarak jauh, dan alur kerja multi-platform.

| Kelebihan | Kekurangan |
|------|------|
| Rasio kompresi-ke-kualitas sangat baik | Encoding sedikit lebih kompleks dari AVI |
| Dukungan perangkat universal | |
| Performa streaming bagus | |

### MOV (QuickTime File Format)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#a855f7" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MOV](https://en.wikipedia.org/wiki/QuickTime_File_Format) dikembangkan oleh Apple dan umum dalam alur kerja pasca-produksi. Beberapa NVR mengekspor ke MOV untuk tujuan pengeditan atau pengarsipan.

**Terbaik untuk:** Alur kerja berbasis macOS dan rekaman yang ditujukan untuk pengeditan video.

| Kelebihan | Kekurangan |
|------|------|
| Kualitas tinggi, mendukung codec ramah editing | Lebih besar dari MP4 |
| Dukungan metadata baik | Dukungan asli lebih sedikit di luar ekosistem Apple |

[MKV](https://en.wikipedia.org/wiki/Matroska) adalah kontainer standar terbuka yang dapat menampung banyak trek audio, aliran subtitle, dan penanda bab dalam satu file.

**Terbaik untuk:** Ekspor multi-kamera atau pengarsipan rekaman dengan metadata.

| Kelebihan | Kekurangan |
|------|------|
| Mendukung trek tak terbatas dalam satu file | Kurang dukungan di software NVR mainstream |
| Sumber terbuka, tanpa biaya lisensi | Tidak kompatibel secara universal |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#3b82f6" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[WMV](https://en.wikipedia.org/wiki/Windows_Media_Video) dirancang oleh Microsoft untuk pemutaran berbasis Windows. Format ini mencapai ukuran file kecil tetapi dengan mengorbankan kualitas yang cukup terlihat.

**Terbaik untuk:** Sistem DVR lawas khusus Windows.

| Kelebihan | Kekurangan |
|------|------|
| Ukuran file kecil | Kualitas lebih rendah dari codec modern |
| Pemutaran asli Windows | Dukungan lintas platform buruk |

**Terbaik untuk:** Antarmuka pengawasan berbasis webcam atau browser lawas.

### H.264 (Advanced Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <text x="18" y="29" fontSize="12" fill="#059669" fontWeight="bold" fontFamily="monospace">264</text>

[H.264](https://en.wikipedia.org/wiki/H.264/MPEG-4_AVC) adalah standar kompresi video yang paling banyak digunakan di CCTV. Format ini dapat mengurangi ukuran file hingga 50% dibandingkan MJPEG sambil mempertahankan kualitas yang sebanding.

**Terbaik untuk:** Rekaman HD dan pengawasan 24/7 berkelanjutan pada perangkat keras yang ada.

| Kelebihan | Kekurangan |
|------|------|
| Dukungan dekoder perangkat keras luas | Terkendala lisensi (meski sebagian besar OEM menanggungnya) |
| Matang, teruji baik dalam pengawasan | Digantikan oleh H.265 |

### H.265 (High Efficiency Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[H.265](https://en.wikipedia.org/wiki/High_Efficiency_Video_Coding) (disebut juga HEVC) kira-kira menggandakan rasio kompresi H.264, menjadikannya pilihan utama untuk NVR 4K dan multi-kanal.

**Terbaik untuk:** Kamera 4K, NVR kanal tinggi, dan sistem dengan penyimpanan atau bandwidth terbatas.

| Kelebihan | Kekurangan |
|------|------|
| File ~50% lebih kecil dari H.264 pada kualitas sama | Membutuhkan daya pemrosesan lebih |
| Ideal untuk resolusi 4K+ | Dukungan terbatas di perangkat keras NVR lama |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MJPEG](https://en.wikipedia.org/wiki/Motion_JPEG) mengkodekan setiap bingkai sebagai gambar JPEG independen. Ini membuat analisis bingkai-per-bingkai sederhana tetapi menghasilkan file besar karena tidak ada kompresi antar-bingkai yang digunakan.

**Terbaik untuk:** Aplikasi yang memerlukan latensi encoding rendah atau ekstraksi bingkai mudah.

| Kelebihan | Kekurangan |
|------|------|
| Setiap bingkai adalah gambar penuh (mudah mengambil gambar diam) | Kompresi buruk dibandingkan H.264/H.265 |
| Latensi encoding rendah | Ukuran file besar untuk perekaman terus-menerus |

### RAW / Video Tak Terkompresi

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[Video tak terkompresi](https://en.wikipedia.org/wiki/Uncompressed_video) menangkap setiap piksel dari sensor tanpa kompresi. File berukuran sangat besar tetapi menawarkan nilai bukti maksimal.

**Terbaik untuk:** Bukti forensik, barang bukti pengadilan, dan analisis investigasi.

| Kelebihan | Kekurangan |
|------|------|
| Tanpa artefak kompresi | Kebutuhan penyimpanan sangat besar secara tidak praktis |
| Detail maksimal untuk analisis | Memerlukan perangkat lunak pemutaran khusus |

Banyak produsen DVR menggunakan [format kontainer proprietary](https://en.wikipedia.org/wiki/Proprietary_format) yang menyematkan metadata seperti stempel waktu, peristiwa gerakan, dan ID kamera. File-file ini memerlukan perangkat lunak atau pemutar dari vendor untuk diputar.

**Terbaik untuk:** Pemutaran mulus dalam ekosistem pabrikan.

| Kelebihan | Kekurangan |
|------|------|
| Metadata kaya (waktu, gerakan, ID kamera) | Terikat pada satu vendor |
| Dioptimalkan untuk perangkat keras spesifik | Ekspor biasanya memerlukan konversi |

Pilihan codec secara langsung memengaruhi perencanaan penyimpanan. Sistem 4 kanal 1080p yang merekam 24/7 membutuhkan kira-kira:

| Codec | Penyimpanan Harian (4× 1080p @ 15 fps) |
|-------|-----------------------------------|
| MJPEG | ~300 GB |
| H.264 | ~80 GB |
| H.265 | ~40 GB |
| RAW   | ~2+ TB |

Lihat [implementasi referensi H.264](https://en.wikipedia.org/wiki/X264) atau [proyek x265](https://en.wikipedia.org/wiki/X265) untuk pendalaman teknis tentang codec ini.
