La scelta del formato file giusto per i filmati CCTV influisce sulla qualità video, sui requisiti di archiviazione, sulla compatibilità di riproduzione e sull'usabilità forense. Questa guida copre i principali formati contenitore e codec utilizzati nei sistemi di sorveglianza.

## Contenitori vs Codec

<svg viewBox="0 0 64 64" width="48" height="48" fill="none" stroke="#2563eb" strokeWidth="2" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'12px'}}>

Un [formato contenitore](https://en.wikipedia.org/wiki/Container_format) (come MP4 o AVI) raggruppa video, audio e metadati insieme. Un [codec](https://en.wikipedia.org/wiki/Video_codec) (come H.264 o H.265) determina come il video viene compresso all'interno di quel contenitore. I moderni sistemi CCTV utilizzano tipicamente video H.264 o H.265 all'interno di un contenitore MP4.

### AVI (Audio Video Interleave)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#6366f1" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <rect x="34" y="36" width="8" height="4" rx="1" fill="#6366f1" opacity="0.2"/>
</svg>

[AVI](https://en.wikipedia.org/wiki/Audio_Video_Interleave) è stato introdotto da Microsoft nel 1992. Memorizza i dati con una compressione minima, producendo file di grandi dimensioni con alta qualità.

**Ideale per:** Clip forensi a breve termine dove il dettaglio è più importante dello storage.

| Pro | Contro |
|------|------|
| Alta qualità, ampio supporto di riproduzione | Dimensioni file molto grandi |
| Nessuna codifica complessa necessaria | Scarsa efficienza di compressione |

### MP4 (MPEG-4 Part 14)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#10b981" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <rect x="6" y="6" width="36" height="36" rx="4"/>
  <rect x="10" y="10" width="28" height="28" rx="2" fill="#10b981" opacity="0.1"/>
  <polygon points="20,16 20,32 32,24" fill="#10b981" opacity="0.4"/>
</svg>

[MP4](https://en.wikipedia.org/wiki/MP4_file_format) è il formato dominante nella sorveglianza moderna. Bilancia l'efficienza di compressione con un'ampia compatibilità su NVR, app mobili e piattaforme cloud.

**Ideale per:** Registrazione a lungo termine, visualizzazione remota e flussi di lavoro multipiattaforma.

| Pro | Contro |
|------|------|
| Eccellente rapporto compressione-qualità | Codifica leggermente più complessa dell'AVI |
| Supporto universale dei dispositivi | |
| Buone prestazioni di streaming | |

### MOV (QuickTime File Format)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#a855f7" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MOV](https://en.wikipedia.org/wiki/QuickTime_File_Format) è stato sviluppato da Apple ed è comune nei flussi di post-produzione. Alcuni NVR esportano in MOV per scopi di editing o archiviazione.

**Ideale per:** Flussi di lavoro basati su macOS e filmati destinati al montaggio video.

| Pro | Contro |
|------|------|
| Alta qualità, supporta codec adatti all'editing | Più grande di MP4 |
| Buon supporto dei metadati | Minor supporto nativo al di fuori dell'ecosistema Apple |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#f59e0b" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MKV](https://en.wikipedia.org/wiki/Matroska) è un contenitore open-standard che può contenere multiple tracce audio, flussi di sottotitoli e marcatori di capitolo in un singolo file.

**Ideale per:** Esportazioni multi-telecamera o archiviazione di filmati con metadati.

| Pro | Contro |
|------|------|
| Supporta tracce illimitate in un file | Meno supportato nei software NVR mainstream |
| Open-source, nessuna tassa di licenza | Non universalmente compatibile |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#3b82f6" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[WMV](https://en.wikipedia.org/wiki/Windows_Media_Video) è stato progettato da Microsoft per la riproduzione su Windows. Raggiunge dimensioni file ridotte ma a un notevole costo di qualità.

**Ideale per:** Sistemi DVR legacy solo Windows.

| Pro | Contro |
|------|------|
| Dimensioni file ridotte | Qualità inferiore rispetto ai codec moderni |
| Riproduzione nativa Windows | Scarso supporto multipiattaforma |

**Ideale per:** Interfacce di sorveglianza legacy basate su webcam o browser.

| Pro | Contro |

### H.264 (Advanced Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <circle cx="24" cy="24" r="8"/>

  <text x="18" y="29" fontSize="12" fill="#059669" fontWeight="bold" fontFamily="monospace">264</text>
</svg>

[H.264](https://en.wikipedia.org/wiki/H.264/MPEG-4_AVC) è lo standard di compressione video più diffuso in ambito CCTV. Può ridurre la dimensione dei file fino al 50% rispetto a MJPEG mantenendo una qualità comparabile.

**Ideale per:** Registrazione HD e sorveglianza continua 24/7 su hardware esistente.

| Pro | Contro |
|------|------|
| Ampio supporto di decoder hardware | Vincoli di licenza (anche se la maggior parte degli OEM li copre) |
| Maturo, ben testato nella sorveglianza | In fase di sostituzione da H.265 |

### H.265 (High Efficiency Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <text x="25" y="29" fontSize="10" fill="#059669" fontWeight="bold" fontFamily="monospace">265</text>

[H.265](https://en.wikipedia.org/wiki/High_Efficiency_Video_Coding) (chiamato anche HEVC) raddoppia circa il rapporto di compressione di H.264, rendendolo la scelta ideale per NVR 4K e multi-canale.

**Ideale per:** Telecamere 4K, NVR ad alto numero di canali e sistemi con archiviazione o larghezza di banda limitata.

| Pro | Contro |
|------|------|
| File ~50% più piccoli di H.264 alla stessa qualità | Richiede più potenza di elaborazione |
| Ideale per risoluzioni 4K+ | Supporto limitato in hardware NVR più vecchi |

### MJPEG (Motion JPEG)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

  <circle cx="33" cy="16" r="1.5" fill="#059669"/>

[MJPEG](https://en.wikipedia.org/wiki/Motion_JPEG) codifica ogni fotogramma come un'immagine JPEG indipendente. Questo rende l'analisi fotogramma per fotogramma semplice ma produce file grandi poiché non viene utilizzata alcuna compressione inter-fotogramma.

**Ideale per:** Applicazioni che richiedono bassa latenza di codifica o facile estrazione di fotogrammi.

| Pro | Contro |
|------|------|
| Ogni fotogramma è un'immagine completa (facile estrarre fermi immagine) | Compressione scarsa rispetto a H.264/H.265 |
| Bassa latenza di codifica | Dimensioni file elevate per registrazione continua |

### RAW / Video non compresso

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <polygon points="24,4 44,24 24,44 4,24" fill="#059669" opacity="0.1"/>
  <polygon points="24,10 38,24 24,38 10,24" fill="none" strokeWidth="1" opacity="0.4"/>

  <circle cx="24" cy="24" r="2" fill="#059669" opacity="0.4"/>
</svg>

[Il video non compresso](https://en.wikipedia.org/wiki/Uncompressed_video) cattura ogni pixel dal sensore senza compressione. I file sono enormi ma offrono il massimo valore probatorio.

**Ideale per:** Prove forensi, esposizioni in tribunale e analisi investigative.

| Pro | Contro |
|------|------|
| Nessun artefatto di compressione | Requisiti di archiviazione impraticabilmente grandi |
| Massimo dettaglio per l'analisi | Richiede software di riproduzione specializzato |

Molti produttori di DVR utilizzano [formati contenitore proprietari](https://en.wikipedia.org/wiki/Proprietary_format) che incorporano metadati come timestamp, eventi di movimento e ID telecamera. Questi file richiedono il software o il lettore del vendor per la riproduzione.

**Ideale per:** Riproduzione senza soluzione di continuità all'interno dell'ecosistema del produttore.

| Pro | Contro |
|------|------|
| Metadati ricchi (tempo, movimento, ID telecamera) | Legato a un unico vendor |
| Ottimizzato per l'hardware specifico | L'esportazione richiede solitamente conversione |

La scelta del codec influisce direttamente sulla pianificazione dello storage. Un sistema a 4 canali 1080p che registra 24/7 richiede approssimativamente:

| Codec | Archiviazione giornaliera (4× 1080p @ 15 fps) |
|-------|-----------------------------------|
| MJPEG | ~300 GB |
| H.264 | ~80 GB |
| H.265 | ~40 GB |
| RAW   | ~2+ TB |

Fare riferimento all'[implementazione di riferimento H.264](https://en.wikipedia.org/wiki/X264) o al [progetto x265](https://en.wikipedia.org/wiki/X265) per approfondimenti tecnici su questi codec.

| Flussi macOS o editing video | MOV con H.264 |
