การเลือกรูปแบบไฟล์ที่เหมาะสมสำหรับภาพจาก CCTV ส่งผลต่อคุณภาพวิดีโอ ความต้องการพื้นที่จัดเก็บ ความเข้ากันได้ในการเล่น และการใช้งานทางนิติเวช คู่มือนี้ครอบคลุมรูปแบบคอนเทนเนอร์และโคเดกหลักที่ใช้ในระบบเฝ้าระวัง

### AVI (Audio Video Interleave)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#6366f1" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[AVI](https://en.wikipedia.org/wiki/Audio_Video_Interleave) ถูกนำเสนอโดย Microsoft ในปี 1992 มันเก็บข้อมูลด้วยการบีบอัดน้อยที่สุด ทำให้ได้ไฟล์ขนาดใหญ่ที่มีคุณภาพสูง

**เหมาะสำหรับ:** คลิปนิติเวชระยะสั้นที่รายละเอียดสำคัญกว่าพื้นที่จัดเก็บ

| ข้อดี | ข้อเสีย |
|------|------|
| คุณภาพสูง รองรับการเล่นได้กว้าง | ขนาดไฟล์ใหญ่มาก |
| ไม่ต้องเข้ารหัสซับซ้อน | ประสิทธิภาพการบีบอัดต่ำ |

### MP4 (MPEG-4 Part 14)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#10b981" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MP4](https://en.wikipedia.org/wiki/MP4_file_format) เป็นรูปแบบที่โดดเด่นในระบบเฝ้าระวังสมัยใหม่ มันสร้างสมดุลระหว่างประสิทธิภาพการบีบอัดกับความเข้ากันได้กว้างทั่วทั้ง NVR แอปมือถือ และแพลตฟอร์มคลาวด์

**เหมาะสำหรับ:** การบันทึกระยะยาว การดูระยะไกล และขั้นตอนการทำงานข้ามแพลตฟอร์ม

| ข้อดี | ข้อเสีย |
|------|------|
| อัตราส่วนการบีบอัดต่อคุณภาพดีเยี่ยม | การเข้ารหัสซับซ้อนกว่า AVI เล็กน้อย |
| รองรับอุปกรณ์สากล | |
| ประสิทธิภาพการสตรีมที่ดี | |

[MOV](https://en.wikipedia.org/wiki/QuickTime_File_Format) พัฒนาโดย Apple และพบได้ทั่วไปในขั้นตอนการทำงานหลังการผลิต NVR บางตัวส่งออกเป็น MOV เพื่อการตัดต่อหรือเก็บถาวร

**เหมาะสำหรับ:** ขั้นตอนการทำงานบน macOS และภาพที่ต้องการตัดต่อวิดีโอ

| ข้อดี | ข้อเสีย |
|------|------|

| รองรับข้อมูลเมตาที่ดี | การรองรับแบบเนทีฟน้อยนอกระบบนิเวศของ Apple |

[MKV](https://en.wikipedia.org/wiki/Matroska) เป็นคอนเทนเนอร์มาตรฐานเปิดที่สามารถเก็บแทร็กเสียงหลายรายการ สตรีมคำบรรยาย และเครื่องหมายบทในไฟล์เดียว

**เหมาะสำหรับ:** การส่งออกหลายกล้องหรือการเก็บถาวรภาพพร้อมข้อมูลเมตา

| ข้อดี | ข้อเสีย |
|------|------|

| โอเพนซอร์ส ไม่มีค่าลิขสิทธิ์ | ไม่เข้ากันได้แบบสากล |

**เหมาะสำหรับ:** อินเทอร์เฟซเฝ้าระวังรุ่นเก่าที่ใช้เว็บแคมหรือเบราว์เซอร์

### H.264 (Advanced Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[H.264](https://en.wikipedia.org/wiki/H.264/MPEG-4_AVC) เป็นมาตรฐานการบีบอัดวิดีโอที่ใช้กันอย่างแพร่หลายที่สุดใน CCTV สามารถลดขนาดไฟล์ได้ถึง 50% เมื่อเทียบกับ MJPEG ในขณะที่รักษาคุณภาพที่เทียบเคียงได้

**เหมาะสำหรับ:** การบันทึก HD และการเฝ้าระวังต่อเนื่อง 24/7 บนฮาร์ดแวร์ที่มีอยู่

| ข้อดี | ข้อเสีย |
|------|------|
| รองรับตัวถอดรหัสฮาร์ดแวร์อย่างกว้างขวาง | ข้อจำกัดด้านลิขสิทธิ์ (แม้ OEM ส่วนใหญ่จะครอบคลุม) |
| โตเต็มที่ ทดสอบดีในงานเฝ้าระวัง | กำลังถูกแทนที่โดย H.265 |

### H.265 (High Efficiency Video Coding)

[H.265](https://en.wikipedia.org/wiki/High_Efficiency_Video_Coding) (หรือเรียกอีกอย่างว่า HEVC) เพิ่มอัตราส่วนการบีบอัดของ H.264 ประมาณสองเท่า ทำให้เป็นตัวเลือกที่เหมาะสมที่สุดสำหรับ NVR 4K และหลายช่องสัญญาณ

**เหมาะสำหรับ:** กล้อง 4K, NVR หลายช่องสัญญาณ และระบบที่มีพื้นที่จัดเก็บหรือแบนด์วิดท์จำกัด

| ข้อดี | ข้อเสีย |
|------|------|
| ไฟล์เล็กกว่า H.264 ~50% ในคุณภาพเท่ากัน | ต้องใช้พลังประมวลผลมากขึ้น |
| เหมาะสำหรับความละเอียด 4K+ | รองรับในฮาร์ดแวร์ NVR รุ่นเก่าอย่างจำกัด |

[MJPEG](https://en.wikipedia.org/wiki/Motion_JPEG) เข้ารหัสแต่ละเฟรมเป็นภาพ JPEG อิสระ ทำให้การวิเคราะห์แบบเฟรมต่อเฟรมทำได้ง่าย แต่สร้างไฟล์ขนาดใหญ่เนื่องจากไม่มีการบีบอัดระหว่างเฟรม

**เหมาะสำหรับ:** แอปพลิเคชันที่ต้องการความหน่วงในการเข้ารหัสต่ำหรือการแยกเฟรมที่ง่าย

| ความหน่วงในการเข้ารหัสต่ำ | ขนาดไฟล์ใหญ่สำหรับการบันทึกต่อเนื่อง |

[วิดีโอไม่บีบอัด](https://en.wikipedia.org/wiki/Uncompressed_video) จับภาพทุกพิกเซลจากเซนเซอร์โดยไม่มีการบีบอัด ไฟล์มีขนาดใหญ่มากแต่ให้คุณค่าทางหลักฐานสูงสุด

**เหมาะสำหรับ:** หลักฐานนิติเวช หลักฐานในศาล และการวิเคราะห์สืบสวน

| ข้อดี | ข้อเสีย |
|------|------|
| ไม่มีสิ่งรบกวนจากการบีบอัด | ความต้องการพื้นที่จัดเก็บใหญ่เกินไปในทางปฏิบัติ |
| รายละเอียดสูงสุดสำหรับการวิเคราะห์ | ต้องใช้ซอฟต์แวร์เล่นเฉพาะทาง |

ผู้ผลิต DVR หลายรายใช้[รูปแบบคอนเทนเนอร์เฉพาะ](https://en.wikipedia.org/wiki/Proprietary_format)ที่ฝังข้อมูลเมตาเช่น ประทับเวลา เหตุการณ์การเคลื่อนไหว และ ID กล้อง ไฟล์เหล่านี้ต้องใช้ซอฟต์แวร์หรือเครื่องเล่นของผู้ผลิตในการเล่น

**เหมาะสำหรับ:** การเล่นที่ราบรื่นภายในระบบนิเวศของผู้ผลิต

| ข้อดี | ข้อเสีย |
|------|------|
| ข้อมูลเมตาที่สมบูรณ์ (เวลา การเคลื่อนไหว ID กล้อง) | ผูกติดกับผู้ผลิตเดียว |
| ปรับให้เหมาะสมสำหรับฮาร์ดแวร์เฉพาะ | การส่งออกมักต้องแปลงรูปแบบ |

## ข้อควรพิจารณาเรื่องพื้นที่จัดเก็บ

การเลือกโคเดกส่งผลโดยตรงต่อการวางแผนพื้นที่จัดเก็บ ระบบ 4 ช่องสัญญาณ 1080p ที่บันทึก 24/7 ต้องการพื้นที่โดยประมาณ:

| โคเดก | พื้นที่จัดเก็บต่อวัน (4× 1080p @ 15 fps) |
|-------|-----------------------------------|
| MJPEG | ~300 GB |
| H.264 | ~80 GB |
| H.265 | ~40 GB |
| RAW   | ~2+ TB |

อ้างอิงถึง[การใช้งานอ้างอิง H.264](https://en.wikipedia.org/wiki/X264) หรือ[โครงการ x265](https://en.wikipedia.org/wiki/X265) สำหรับรายละเอียดทางเทคนิคเกี่ยวกับโคเดกเหล่านี้
