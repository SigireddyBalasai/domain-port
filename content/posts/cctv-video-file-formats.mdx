Choosing the right file format for CCTV footage affects video quality, storage requirements, playback compatibility, and forensic usability. This guide covers the major container formats and codecs used in surveillance systems.

## Containers vs Codecs

<svg viewBox="0 0 64 64" width="48" height="48" fill="none" stroke="#2563eb" strokeWidth="2" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'12px'}}>

A [container format](https://en.wikipedia.org/wiki/Container_format) (like MP4 or AVI) wraps video, audio, and metadata together. A [codec](https://en.wikipedia.org/wiki/Video_codec) (like H.264 or H.265) determines how the video is compressed inside that container. Modern CCTV systems typically use H.264 or H.265 video inside an MP4 container.

### AVI (Audio Video Interleave)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#6366f1" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[AVI](https://en.wikipedia.org/wiki/Audio_Video_Interleave) was introduced by Microsoft in 1992. It stores data with minimal compression, producing large files with high quality.

**Best for:** Short-term forensic clips where detail matters more than storage.

| Pros | Cons |
|------|------|
| High quality, wide playback support | Very large file sizes |
| No complex encoding needed | Poor compression efficiency |

### MP4 (MPEG-4 Part 14)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#10b981" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>
  <rect x="6" y="6" width="36" height="36" rx="4"/>

</svg>

[MP4](https://en.wikipedia.org/wiki/MP4_file_format) is the dominant format in modern surveillance. It balances compression efficiency with broad compatibility across NVRs, mobile apps, and cloud platforms.

**Best for:** Long-term recording, remote viewing, and multi-platform workflows.

| Pros | Cons |
|------|------|
| Excellent compression-to-quality ratio | Slightly more complex encoding than AVI |
| Universal device support | |
| Good streaming performance | |

### MOV (QuickTime File Format)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#a855f7" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[MOV](https://en.wikipedia.org/wiki/QuickTime_File_Format) was developed by Apple and is common in post-production workflows. Some NVRs export to MOV for editing or archival purposes.

**Best for:** macOS-based workflows and footage destined for video editing.

| Pros | Cons |
|------|------|
| High quality, supports edit-friendly codecs | Larger than MP4 |
| Good metadata support | Less native support outside Apple ecosystem |

[MKV](https://en.wikipedia.org/wiki/Matroska) is an open-standard container that can hold multiple audio tracks, subtitle streams, and chapter markers in a single file.

**Best for:** Multi-camera exports or archiving footage with metadata.

| Pros | Cons |
|------|------|
| Supports unlimited tracks in one file | Less support in mainstream NVR software |
| Open-source, no licensing fees | Not universally compatible out of the box |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#3b82f6" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[WMV](https://en.wikipedia.org/wiki/Windows_Media_Video) was designed by Microsoft for Windows-based playback. It achieves small file sizes but at a noticeable quality cost.

**Best for:** Legacy Windows-only DVR systems.

| Pros | Cons |
|------|------|

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#ef4444" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

| Pros | Cons |
|------|------|
| Small files, fast streaming | Obsolete technology |

### H.264 (Advanced Video Coding)

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[H.264](https://en.wikipedia.org/wiki/H.264/MPEG-4_AVC) is the most widely deployed video compression standard in CCTV. It can reduce file size by up to 50% compared to MJPEG while maintaining comparable quality.

**Best for:** HD recording and continuous 24/7 surveillance on existing hardware.

| Pros | Cons |
|------|------|
| Broad hardware decoder support | Licensing encumbered (though most OEMs cover this) |
| Mature, well-tested in surveillance | Being superseded by H.265 |

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[H.265](https://en.wikipedia.org/wiki/High_Efficiency_Video_Coding) (also called HEVC) roughly doubles the compression ratio of H.264, making it the go-to choice for 4K and multi-channel NVRs.

**Best for:** 4K cameras, high-channel-count NVRs, and systems with limited storage or bandwidth.

| Pros | Cons |
|------|------|
| ~50% smaller files than H.264 at same quality | Requires more processing power |
| Ideal for 4K+ resolutions | Limited support in older NVR hardware |

[MJPEG](https://en.wikipedia.org/wiki/Motion_JPEG) encodes each frame as an independent JPEG image. This makes frame-by-frame analysis simple but produces large files since no inter-frame compression is used.

**Best for:** Applications requiring low encoding latency or easy frame extraction.

| Pros | Cons |
|------|------|
| Each frame is a full image (easy to grab stills) | Poor compression compared to H.264/H.265 |
| Low encoding latency | Large file sizes for continuous recording |

### RAW / Uncompressed Video

<svg viewBox="0 0 48 48" width="36" height="36" fill="none" stroke="#059669" strokeWidth="1.8" strokeLinecap="round" strokeLinejoin="round" style={{float:'left', marginRight:'10px', marginTop:'4px'}}>

[Uncompressed video](https://en.wikipedia.org/wiki/Uncompressed_video) captures every pixel from the sensor with zero compression. Files are enormous but offer maximum evidentiary value.

**Best for:** Forensic evidence, courtroom exhibits, and investigative analysis.

| Pros | Cons |
|------|------|
| No compression artifacts | Impractically large storage requirements |
| Maximum detail for analysis | Requires specialised playback software |

Many DVR manufacturers use [proprietary](https://en.wikipedia.org/wiki/Proprietary_format) container formats that embed metadata such as timestamps, motion events, and camera IDs. These files require the vendor's software or player for playback.

**Best for:** Seamless playback within the manufacturer's ecosystem.

| Pros | Cons |
|------|------|
| Rich metadata (time, motion, camera ID) | Tied to one vendor |
| Optimised for the specific hardware | Export usually requires conversion |

## Storage Considerations

Codec choice directly affects storage planning. A 4-channel 1080p system recording 24/7 requires roughly:

| Codec | Daily Storage (4× 1080p @ 15 fps) |
|-------|-----------------------------------|
| MJPEG | ~300 GB |
| H.264 | ~80 GB |
| H.265 | ~40 GB |
| RAW   | ~2+ TB |

Refer to the [H.264 reference implementation](https://en.wikipedia.org/wiki/X264) or [x265 project](https://en.wikipedia.org/wiki/X265) for technical depth on these codecs.
